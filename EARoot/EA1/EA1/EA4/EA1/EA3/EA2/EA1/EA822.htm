Las matrices muestran las relaciones principales entre entidades de modelo relacionadas.<br/><br/>Las matrices forman la materia prima para el desarrollo de diagramas y también actúan como un recurso clave para la evaluación de impacto.<br/><br/>En esta etapa, se podría producir una matriz de entidad a aplicaciones para validar esta asignación. Ahora comenzará a entenderse cómo se crean, mantienen, transforman y pasan los datos a otras aplicaciones, o cómo los utilizan otras aplicaciones. Las brechas obvias, como las entidades que nunca parecen ser creadas por una aplicación o los datos creados pero nunca utilizados, deben tenerse en cuenta para un análisis posterior de las brechas.<br/><br/>El inventario de datos racionalizado se puede utilizar para actualizar y refinar los diagramas arquitectónicos de cómo los datos se relacionan con otros aspectos de la arquitectura.<br/><br/>Una vez que se hayan realizado estas actualizaciones, puede ser apropiado incluir una breve iteración de la arquitectura de la aplicación para resolver los cambios identificados.<br/><br/>Las siguientes matrices deben considerarse para el desarrollo dentro de una arquitectura de datos:<br/><br/><ul>
<li>Entidad de datos/función del negocio (que muestra qué datos admiten qué funciones y qué función del negocio posee qué datos)</li><li>Servicio de Negocio/Información (desarrollado durante la fase de Arquitectura del Negocio)</li><li>Aplicación/Datos (desarrollados a lo largo de las fases de Arquitectura de Aplicación y Arquitectura de Datos)</li></ul>
<br/>La estructura de las matrices se basa en los atributos de las entidades metamodelo, tal como se definen en la<i> Parte IV, 30. Metamodelo de contenido</i> .<br/>
